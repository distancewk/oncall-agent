# SuperBizAgent Agent 高频面试题与参考答案

本文基于当前 SuperBizAgent 项目整理，重点覆盖 Agent 架构、AIOps 诊断流程、工具调用、RAG、记忆机制、MCP、可靠性和工程落地。适合用于项目复盘、面试准备和技术讲解。

## 一、项目与 Agent 架构

### 1. 这个项目主要解决什么问题？

SuperBizAgent 是一个智能 OnCall / AIOps 助手，目标是把“告警接收、事故归并、证据查询、AI 诊断、报告生成、历史复用”串成一条自动化流程。

普通用户可以用它做知识库问答和日常聊天；运维场景下，系统接收 Alertmanager 告警后，会归并成 Incident，并触发 DiagnosisRun，让 AI 调用指标、日志、知识库和相似历史案例等工具生成诊断报告。

### 2. 系统整体可以分成哪些层？

可以分成五层：

- 接入层：前端页面、聊天接口、告警 Webhook、Incident 管理接口。
- Agent 编排层：普通聊天 ReactAgent、AIOps Supervisor、Planner、Executor。
- 工具层：时间、内部知识库、Prometheus、日志、MCP Tavily、MCP DBHub。
- 知识与记忆层：Milvus 向量库、文档 RAG、私人记忆、相似历史故障案例。
- 可靠性层：熔断、依赖健康状态、异步任务状态、证据链和报告约束。

### 3. 为什么用 Agent，而不是写固定诊断脚本？

固定脚本适合确定性流程，比如“CPU 高就查 top 进程”。但真实故障排查通常需要根据上下文动态决定下一步：可能先查指标趋势，也可能先查日志、runbook、历史案例或数据库状态。

Agent 的价值在于可以根据告警上下文、工具返回和证据缺口动态规划下一步。但 Agent 不能无限自由，所以项目用提示词、工具边界、证据 id 和报告校验来约束它。

### 4. 普通聊天和 AIOps 诊断有什么区别？

普通聊天是轻量助手，主要使用：

- `getCurrentDateTime`
- `queryInternalDocs`
- Tavily MCP，用于公开互联网资料查询

AIOps 诊断是事故排查流程，会使用更重的诊断工具：

- `queryPrometheusAlerts`
- `queryMetricTrend`
- `queryLogs`
- `getAvailableLogTopics`
- 内部知识库
- 相似历史故障召回
- MCP Tavily / DBHub

这样做的原因是减少普通聊天误调用监控、日志和数据库工具的概率，也降低工具列表过长导致模型选择混乱的问题。

### 5. 多 Agent 编排是怎么设计的？

AIOps 使用 Supervisor + Planner + Executor 的结构：

- Supervisor：负责调度 Planner 和 Executor。
- Planner：负责拆解问题、规划诊断步骤、根据反馈重新规划，并输出最终报告。
- Executor：负责执行 Planner 指定的具体动作，比如调用指标趋势、日志或知识库工具。

这种设计把“想清楚怎么查”和“实际执行查询”拆开，便于控制诊断流程，也便于在报告中要求引用 evidence id。

### 6. Planner 和 Executor 分别承担什么职责？

Planner 关注“诊断策略”：

- 当前事故的主要风险是什么？
- 需要哪些证据才能判断根因？
- 哪些工具应该先调用？
- 证据不足时如何说明？
- 最终报告如何组织？

Executor 关注“执行动作”：

- 按 Planner 要求调用工具。
- 返回工具查询结果。
- 不擅自扩展结论。
- 把证据反馈给 Planner。

### 7. 工具是如何接入 Agent 的？

项目里有两类工具：

- 本地方法工具：通过 Spring AI 的 `@Tool` 暴露，例如时间、知识库、Prometheus、日志工具。
- MCP 工具：通过 Spring AI MCP Client 动态接入，例如 Tavily 和 DBHub。

普通聊天通过 `ChatService` 创建 ReactAgent；AIOps 通过 `AiOpsService` 创建 Planner 和 Executor。两者现在使用不同的工具列表，避免工具面过大。

### 8. 为什么要收敛普通聊天工具列表？

工具越多，模型越容易误选工具。普通聊天如果暴露 Prometheus、日志和数据库工具，用户问一个普通问题时模型也可能尝试查监控或数据库，造成无意义延迟和安全风险。

收敛后，普通聊天只保留时间、知识库和 Tavily；事故诊断才保留完整诊断工具链。

## 二、Incident 与诊断流程

### 9. 告警进入系统后的流程是什么？

典型流程是：

1. Alertmanager 通过 Webhook 推送告警。
2. 后端根据告警标签生成 aggregationKey。
3. 相同 aggregationKey 的告警归并到同一个 Incident。
4. 系统创建一个新的 DiagnosisRun。
5. 诊断任务异步执行。
6. Agent 调用指标、日志、知识库和相似案例等工具。
7. 每次工具调用保存为 evidence。
8. 最终报告写入 DiagnosisRun.report。

### 10. Incident 和 DiagnosisRun 的区别是什么？

Incident 表示“一次故障事件”，例如某个服务 CPU 长时间过高。

DiagnosisRun 表示“对这个故障做的一次 AI 分析”。一个 Incident 可以有多个 DiagnosisRun，例如首次自动诊断一份报告，人工点击重新诊断又生成一份新报告。

### 11. 一个 Incident 可以存几份诊断报告？

没有硬编码数量上限。

一个 Incident 内部维护 `diagnosisRuns` 列表；每个 DiagnosisRun 最多保存一份最终报告。所以一个 Incident 可以存 N 份报告，N 等于该 Incident 下已完成且带有 `report` 的 DiagnosisRun 数量。

### 12. DiagnosisRun 有哪些状态？

当前主要状态包括：

- `QUEUED`：任务已创建，等待执行。
- `RUNNING`：诊断正在运行。
- `WAITING_TOOL`：正在等待工具调用结果。
- `COMPLETED`：诊断完成并生成报告。
- `FAILED`：诊断失败。
- `CANCELLED`：预留的取消状态。

前端通过这些状态展示“正在查指标、正在查日志、正在匹配 runbook、已完成、失败”等过程。

### 13. 为什么需要诊断证据链？

没有证据链，AI 报告很容易变成“看起来合理但不可追溯”的文本。

项目把每次工具调用保存成 DiagnosisEvidence，包括工具名、参数、摘要、原始片段、是否成功、错误码等。最终报告必须引用 evidence id，这样用户可以追溯结论来自哪次指标查询、日志查询或知识库检索。

### 14. 项目如何降低诊断报告幻觉？

主要靠四层约束：

- 提示词要求：禁止编造数据，结论必须绑定 evidence id。
- 工具证据记录：工具返回结果持久化为 evidence。
- 报告后处理：过滤不可用证据、标记证据缺失、追加风险提示。
- 熔断语义：依赖不可用时返回 `CIRCUIT_OPEN` 或 `DEPENDENCY_ERROR`，报告必须说明证据缺失，不能虚构趋势或日志。

### 15. 既然已经收到告警，为什么还可能查询 Prometheus 告警？

收到的 Webhook 是“当前告警上下文”的首要事实源，一般不应该把查询 Prometheus 活跃告警作为默认第一步。

但在以下情况可以查 Prometheus：

- 需要确认是否还有其它相关活跃告警。
- Webhook 上下文缺少关键标签。
- 需要判断告警是否已经恢复。
- 需要做跨服务关联。

项目提示词已经约束：有明确 Incident 上下文时，不要默认重复查询活跃告警。

### 16. 指标趋势查询工具解决什么问题？

只看当前告警容易误判。比如 CPU 当前 95%，可能是持续上升，也可能只是瞬时尖峰。

`queryMetricTrend` 会查询 15m / 1h / 6h 的趋势，返回 min、max、avg、latest、方向和是否异常。诊断 CPU、内存、错误率、P99、重启次数时，趋势证据比单点值更可靠。

### 17. 获取日志主题工具有什么作用？

`getAvailableLogTopics` 是日志主题发现工具，用于模型不知道该查哪个日志主题时。

如果告警类型已经能推断主题，就应该直接查 `queryLogs`。例如 CPU 或内存问题通常查 `system-metrics`，服务不可用通常查 `application-logs` 或 `system-events`。只有无法推断时才调用主题列表，避免工具链冗长。

### 18. 相似历史故障召回有什么价值？

相似历史故障可以提供“曾经怎么处理”的经验，包括已确认根因、处理动作和复盘结论。

新故障来时先召回相似案例，可以提升报告稳定性，减少模型只根据当前告警描述猜测根因。项目会把已确认故障报告写入知识库，并用相似检索作为诊断上下文。

## 三、RAG 与记忆机制

### 19. 知识库检索是如何工作的？

文档上传后会被切分成 chunk，生成 embedding，写入 Milvus。查询时先做向量召回，再结合 rerank 返回最相关片段。

知识库主要服务两个场景：

- 普通聊天：查询 runbook、技术指南、内部流程。
- AIOps 诊断：匹配故障处理手册、历史案例和操作建议。

### 20. 为什么要做 RAG 数据隔离？

知识库里可能同时存在普通文档、私人记忆、历史故障案例。如果不隔离，普通知识库检索可能串出用户私人记忆，或者不同会话之间互相污染。

项目通过 metadata 里的 `doc_type`、`_source`、`session_id` 区分数据：

- 普通知识库只查文档数据。
- 私人记忆必须按 `session_id` 过滤。
- 历史故障案例使用单独 doc_type。

### 21. 私人记忆摘要是如何筛选的？

私人记忆不是把每轮聊天原文都写入向量库，而是由大模型从对话里抽取长期有价值的信息。

适合进入私人记忆的信息包括：

- 用户偏好，例如希望中文回答、喜欢简洁结论。
- 稳定事实，例如项目名称、技术栈、长期目标。
- 长期约束，例如不要使用某类部署方式。

不适合进入记忆的信息包括：

- 临时任务状态。
- 一次性问题。
- 敏感凭证。
- 不确定或明显过期的信息。

### 22. 为什么之前重启后没有历史信息？

原先如果只把短期会话放在内存或 Redis 里，重启后就会丢失。现在项目把完整聊天历史持久化到文件存储，同时把长期私人记忆写入向量库。

可以理解为：

- 聊天历史：用于还原会话原文。
- 私人记忆：用于跨会话提取稳定偏好和事实。

### 23. 短期记忆为什么还需要过期时间？

短期记忆适合保存当前会话上下文，但不应该无限堆积，否则会带来成本、噪声和隐私风险。

长期有价值的信息应该进入私人记忆摘要；完整历史则通过持久化文件保存。这样既能恢复历史，又不会把所有历史都塞进每次模型上下文。

## 四、MCP、外部工具与安全边界

### 24. 当前配置了哪些 MCP？

当前主要配置了两个 MCP：

- Tavily MCP：用于联网搜索公开互联网资料、官方文档、错误码和版本差异。
- DBHub MCP：用于多数据库只读查询，适合查 schema 或业务数据。

默认不启用 MCP。需要 Tavily 时启用 `mcp` profile；需要数据库查询时额外启用 `mcp-db` profile，并提供真实 DBHub 配置。

### 25. Tavily MCP 什么时候使用？

Tavily 用于公开互联网信息，不应该替代内部事实源。

适合场景：

- 查询公开官方文档。
- 查询错误码含义。
- 查询版本差异。
- 查询天气等公开信息。

不适合场景：

- 内部告警事实。
- 内部日志。
- 内部知识库。
- 数据库业务数据。

### 26. DBHub MCP 什么时候使用？

DBHub 用于数据库只读查询，例如：

- 查看表结构。
- 查询某段时间内的业务数据。
- 排查订单、任务、配置等业务状态。

它不应该默认暴露给普通聊天。只有当用户明确提出查数据库、表结构或业务数据，且系统处于允许数据库 MCP 的场景时，才应该使用。

### 27. MCP 失败为什么不能影响项目启动？

MCP 是外部能力，可能因为 npm 包下载失败、API key 缺失、DBHub 配置错误、数据库连接失败而不可用。如果 MCP 初始化失败直接导致 Spring Boot 启动失败，会影响核心聊天和告警接收能力。

项目引入熔断和依赖隔离后，外部依赖失败应体现为工具不可用或证据缺失，而不是整个应用崩溃。

### 28. 为什么数据库 MCP 必须只读？

Agent 调用数据库工具存在误操作风险。如果允许写 SQL，模型可能因为理解错误执行 `UPDATE`、`DELETE`、`DROP` 等危险操作。

所以数据库 MCP 只能使用只读账号，并且提示词明确禁止写入和结构变更语句。真实生产环境还应在数据库权限层面限制，而不是只依赖提示词。

## 五、可靠性与工程落地

### 29. 项目为什么要设计熔断机制？

这个项目依赖很多外部系统：DashScope、Milvus、Prometheus、CLS、Tavily、DBHub。任何一个依赖不可用都不应该导致系统无限等待或生成虚假报告。

熔断机制的目标是：

- 快速失败。
- 避免雪崩。
- 前端能看到依赖状态。
- 报告能明确说明证据缺失。
- 禁止在依赖失败时编造内容。

### 30. 熔断后为什么不能生成虚构内容？

如果 Prometheus 或日志系统熔断，系统就没有对应证据。此时报告只能说“趋势证据缺失”或“日志证据缺失”，不能继续写“CPU 持续上升”“日志显示大量 500”之类结论。

项目通过 evidence 的 `errorCode` 和报告约束逻辑，把失败证据从可引用证据里排除，并在报告里追加风险提示。

### 31. 为什么 Milvus 要懒加载？

如果应用启动时强依赖 Milvus，Milvus 未启动或网络不可达会导致整个 Spring Boot 启动失败。

懒加载后，核心应用可以先启动，只有实际执行向量检索、上传索引或记忆写入时才初始化 Milvus 客户端。这样本地开发和部分功能验证更稳定。

### 32. 上传知识库为什么改成异步索引？

文件上传和向量索引是两个不同阶段。向量化、批量 insert、collection load 都可能耗时或失败。

异步索引后，上传接口只表示“文件已保存并提交索引任务”，前端可以看到 `INDEXING`、`COMPLETED`、`FAILED` 状态，避免用户误以为上传成功就一定已经可检索。

### 33. 为什么要批量生成 embedding 和批量 insert？

逐条生成 embedding、逐条 insert Milvus 会造成大量网络往返，吞吐低且更容易触发限流。

批量处理可以：

- 降低 API 调用次数。
- 提高索引速度。
- 减少 Milvus 写入开销。
- 便于统一处理部分失败或批大小限制。

### 34. 前端为什么需要展示诊断进度？

诊断可能涉及多次工具调用，耗时较长。如果前端只显示“处理中”，用户无法判断系统是否卡住。

展示 `QUEUED / RUNNING / WAITING_TOOL / COMPLETED / FAILED`，以及当前步骤和当前工具，可以让用户知道系统正在查指标、查日志、匹配 runbook 还是生成报告。

### 35. 自动刷新为什么不能直接重渲染整个详情页？

如果轮询时每次清空 DOM、显示 loading、重置滚动条，会造成页面跳动，用户正在看的内容会被顶回顶部。

正确做法是区分首次加载和后台刷新：

- 首次加载可以显示 loading 并滚动到顶部。
- 后台轮询不显示 loading，不重置滚动位置。
- 如果状态已完成或失败，停止轮询。
- 数据没有变化时跳过 DOM 重绘。

## 六、可继续优化的问题

### 36. 当前 Agent 编排还可以怎么优化？

可以继续做：

- 诊断计划结构化输出，而不是纯自然语言计划。
- 工具选择策略显式化，例如先趋势、再日志、再 runbook。
- 对不同告警类型建立诊断 playbook。
- 为 Planner 增加反思步骤，检查证据缺口。
- 把最终报告拆成结构化字段：根因、影响面、证据、处理动作、风险。

### 37. 当前 RAG 还可以怎么优化？

可以继续做：

- 文档质量评分。
- 召回结果去重。
- 按文档类型设置不同 topK。
- 对 runbook、历史案例、私人记忆使用不同索引或过滤策略。
- 在前端展示命中文档和相似度，便于调试“为什么搜不到”。

### 38. 当前故障诊断还缺哪些企业级能力？

可以补充：

- 服务拓扑依赖图。
- Trace 查询。
- 发布变更关联。
- Kubernetes 事件查询。
- 自动生成修复建议和回滚建议。
- 人工确认根因后沉淀为历史案例。
- 多租户和权限隔离。

### 39. 面试中如何概括这个项目的亮点？

可以这样回答：

这个项目不是单纯聊天机器人，而是一个围绕 Incident 的 AIOps Agent 系统。它把告警归并、诊断运行、工具证据、RAG、私人记忆、相似历史故障和熔断机制结合起来。核心设计重点不是“让模型自由发挥”，而是通过工具边界、证据链和依赖状态约束模型，让诊断报告可追溯、可复现、可失败而不编造。

# 内存溢出/进程OOM告警处理方案

## 告警名称
- **告警名**: `OOMKilled`
- **告警级别**: 严重
- **触发条件**: 进程或容器因内存不足被系统OOM Killer终止

## 问题描述
当进程或容器因内存使用超过限制被系统强制终止时，可能导致：
- 服务实例不可用
- 请求失败率突增
- 数据丢失风险（未持久化的数据）
- 业务中断

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询系统事件日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `system-events`
- **时间范围**: 最近30分钟
- **查询条件**: `OOM OR out_of_memory OR killed OR exit_code:137`

**查询示例**:
```
地域: ap-guangzhou
日志主题: system-events
时间范围: 2024-01-20 14:00:00 到 2024-01-20 14:30:00
查询语句: OOM AND (process_name:java OR process_name:node)
```

### 步骤3: 查询应用错误日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 最近30分钟
- **查询条件**: `OutOfMemoryError OR cannot_allocate OR memory_limit OR java.lang.OutOfMemoryError`

### 步骤4: 查询内存使用指标
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `system-metrics`
- **查询条件**: `memory_usage:>90% OR heap_usage:>90% OR container_memory:>limit`

## 常见原因分析

### 原因1: JVM堆内存配置过小
**特征**:
- 启动参数中-Xmx设置不合理
- 业务高峰期内存使用率持续走高
- GC频率明显增加
- 堆内存监控显示到达上限

**处理方案**:
1. 调整JVM堆内存参数（-Xmx, -Xms）
2. 根据业务量计算合理的内存配置
3. 重启服务使新配置生效
4. 增加内存使用量监控告警

### 原因2: 容器内存Limit设置过低
**特征**:
- 容器被OOM Killer终止（exit code 137）
- 内存使用达到容器Limit上限
- 同一宿主机其他容器运行正常
- 重启后短时间内再次OOM

**处理方案**:
1. 适当增加容器内存Limit
2. 检查是否设置了合理的Resource Request
3. 评估业务实际内存需求
4. 考虑垂直扩容或增加副本数

### 原因3: 内存泄漏
**特征**:
- 内存使用随时间持续增长，不会回落
- GC无法回收的对象越来越多
- Full GC频繁触发（FGC次数持续增加）
- 堆转储分析发现有大量相同实例

**处理方案**:
1. 立即重启服务恢复（临时措施）
2. 使用jmap/jstack生成堆转储快照
3. 分析堆转储定位泄漏对象
4. 修复代码中的资源未释放或对象引用问题

### 原因4: 突发流量导致内存飙升
**特征**:
- OOM发生在流量尖峰时刻
- 请求量短时间内大幅增加
- 内存曲线与流量曲线正相关
- 每个请求消耗内存较多

**处理方案**:
1. 临时扩容（增加实例数或提升规格）
2. 启用限流保护，防止请求堆积
3. 检查是否有大对象创建（如文件上传、大查询）
4. 优化请求处理的内存使用效率

## 紧急处理措施

### 立即操作（5分钟内）
1. **重启**: 立即重启被OOM杀死的服务实例
2. **扩容**: 增加服务实例数量分担负载
3. **限流**: 启用限流机制保护服务稳定

### 短期措施（30分钟内）
1. 检查内存配置参数是否合理
2. 分析日志确认OOM根因
3. 调整资源配额（JVM参数、容器Limit）
4. 检查是否有代码级别的内存泄漏

### 长期优化
1. 建立内存使用基线监控
2. 定期进行堆转储分析
3. 代码审查，修复潜在内存泄漏
4. 完善压测流程，上线前模拟高负载场景

## 验证步骤
1. 确认服务恢复正常运行
2. 检查内存使用率在合理范围（<70%）
3. 验证业务功能完整性
4. 观察30分钟确认无再次OOM
5. 检查GC频率和堆使用趋势

## 相关告警
- `HighMemoryUsage`: 内存使用率过高
- `HighCPUUsage`: CPU使用率过高
- `ContainerRestart`: 容器重启次数过多
- `ServiceUnavailable`: 服务不可用
